# 考试提分系统项目总协作规范（最新版）

## 1. 项目总目标

这个项目的根本目标不是做一个花哨系统，而是：

**通过构建实用、稳定、可持续迭代的考试辅助系统，最终提升分数。**

以后所有判断标准，优先级统一如下：

1. 是否有助于提分
2. 是否能真正落地使用
3. 是否减少时间消耗和重复劳动
4. 是否稳定、易维护、可继续扩展
5. 界面、技术方案、AI能力都排在后面

结论：

**系统是手段，提分是目的。**

## 2. 项目推进总原则

### 2.1 先整体方案，后局部实现
以后默认流程：

**先给整体理解 → 再拆模块 → 再定当前阶段目标 → 再明确边界与验收 → 再实现 → 再自测 → 再交付 → 再更新文档**

### 2.2 默认按真实使用场景理解需求
这个系统是长期备考工具，不是普通 demo。以后做功能时默认先考虑：

- 日常怎么用
- 功能在哪个页面最顺手
- 是否符合原有操作习惯
- 是否和已有功能衔接自然
- 会不会增加额外操作成本

### 2.3 有歧义时，以最接近当前工作流的方案推进
默认优先顺序：

1. 和现有系统最一致
2. 和之前确认过的方案最一致
3. 最少打断现有使用路径
4. 最容易后续扩展
5. 先给可用第一版，不做过度设计

### 2.4 默认保留已确认设计基线
凡是已经确认过的东西，后续都应当当作基线，而不是反复推翻。包括但不限于：

- 页面布局
- 按钮位置
- 功能入口
- 数据结构
- 交互方式
- 命名规则
- 版本方向
- 模块边界

### 2.5 系统优先于扩展，提分目标高于系统本身
当前阶段默认优先完成并优化系统本身。系统做好后，再继续大规模录入笔记，再进入真正提分阶段。

同时始终牢记：

**系统只是辅助工具，核心目标始终是考试提分。后续建议与改造应优先服务提分效率，而不是为了系统而系统。**

## 3. 标准需求推进流程

### 第一步：需求对齐

每次需求先整理清楚以下内容：

#### 3.1 真正想解决的问题
不只看表面功能，要看背后的提分目标。

#### 3.2 功能所属模块
建议固定模块：

- 题库与做题
- 错题管理
- 知识库
- 学习记录
- 统计分析
- 画布与交互辅助
- 自动化流程
- AI辅助能力
- 系统配置与数据管理

#### 3.3 输入、处理、输出
每个需求都要明确：

- 输入是什么
- 用户怎么触发
- 系统怎么处理
- 最终输出什么结果
- 数据是否保存
- 下次是否可恢复

#### 3.4 入口位置与联动关系
必须一起说明：

- 页面入口在哪
- 和原来什么按钮并列
- 触发后是弹层、覆盖、跳转还是侧栏
- 会不会影响移动端
- 会不会影响现有功能

#### 3.5 第一版边界
明确：

- 第一版必须有的
- 第一版不要做的
- 后续可扩展的
- 暂时明确排除的

#### 3.6 验收标准
至少包括：

- 功能是否可用
- 是否符合原有操作逻辑
- 是否符合确认过的设计方案
- 数据是否正确保存
- 刷新/重进是否恢复
- 是否兼容移动端
- 是否没有破坏旧功能
- 是否命中当前运行入口
- 用户当前页面是否能直接看到效果

### 第二步：转成实现任务单

#### 任务单模板

**任务名称**  
一句话描述当前要做什么

**任务目标**  
这个功能解决什么问题

**适用模块**  
属于哪个模块

**用户操作路径**  
用户从哪里进入，怎么使用

**功能范围（本次）**  
本次一定做哪些

**不做范围（本次）**  
这次不做哪些，避免误会

**交互规则**  
按钮、弹层、覆盖、保存、恢复等规则

**数据规则**  
保存什么，怎么恢复，是否独立存储

**兼容要求**  
PC / 移动端 / 原页面兼容性要求

**运行入口**  
当前在线页面由哪个入口驱动

**本次修改入口**  
本次具体修改哪套前端或哪条链路

**页面验收入口**  
最终去哪一套页面验收效果

**验收标准**  
完成后如何判断合格

### 第三步：实现默认规则

#### 4.1 优先在现有基础上改，不随意推翻旧结构
除非明确说“整版替换”，否则默认：

- 先复用现有页面
- 先保留原有入口
- 先兼容已有逻辑
- 在原结构上增量修改

#### 4.2 样式和布局优先向原版对齐
默认要求：

- 按钮位置先对齐
- 结构层级先对齐
- 操作习惯先对齐
- 再做额外优化

#### 4.3 第一版先追求完整可用
第一版优先保证：

- 主路径完整
- 核心功能打通
- 数据可保存
- 可恢复
- 不破坏旧功能
- 落在当前运行入口

#### 4.4 每做一步都要自测
交付前默认检查：

- 是否有语法错误
- 是否有重复声明
- 是否有功能入口失效
- 是否有样式错位
- 是否有移动端兼容问题
- 是否有旧功能被破坏
- 是否有数据保存/恢复异常
- 是否与任务单一致
- 是否改到了当前运行入口
- 当前页面是否能看见改动

#### 4.5 交付的是可直接使用的结果
以后优先交付：

- 可直接替换的文件
- 补丁包
- 完整版本包
- 明确变更说明
- 明确测试说明
- 需要时同步交付交互图、结构示意图、版本记录

#### 4.6 前端入口识别规则（新增）
以后凡涉及前端页面改造，默认先做下面 3 个判断：

1. 当前正在运行的入口是什么
2. 本次准备修改的入口是什么
3. 这两个入口是不是同一个

若不是同一个，必须先明确：

- 本次是要改“当前运行壳”还是“主源码链路”
- 当前页面会不会立刻生效
- 是否还需要做同步回灌

默认优先级：

1. 当前运行壳
2. 当前页面验收壳
3. 主源码同步链路

## 4. 版本推进规则

以后按四层推进：

### A. 项目层
整个系统当前阶段目标是什么

### B. 模块层
本阶段重点模块是什么

### C. 功能层
模块里当前只推进哪个功能

### D. 细节层
按钮颜色、提示文案、动画、图标、微交互等

默认要求：

- 不要只盯细节，不交代整体
- 不要只做单点功能，不说明它在整个系统中的位置
- 不要碎片化推进到失去主线

## 5. 项目沟通规则

### 5.1 先复述真实意图，不只复述字面需求
### 5.2 遇到多个理解方向时，选最保守且最贴近已有方案的
### 5.3 用户确认过的内容优先级最高

优先级顺序：

1. 用户明确确认过的方案
2. 已沉淀的项目规则
3. 当前系统现状
4. 我的优化建议

### 5.4 先解决主问题，再说优化项
### 5.5 不要只讲思路，要尽量交付可直接使用的结果

## 6. 页面交互与布局类需求规则

凡涉及以下内容时，默认不能只用文字描述，必须同步给出对应的交互图或结构示意图：

- 页面布局调整
- 按钮位置变化
- 新入口增加
- 弹层 / 抽屉 / 覆盖层 / 跳转页
- 模块之间的流转关系
- PC 与移动端的交互差异
- 原页面改造类需求

交互图至少要表达清楚：

1. 入口在哪
2. 点击后发生什么
3. 页面是否跳转
4. 是弹层、覆盖还是内嵌展开
5. 与原有功能的相对位置关系
6. 用户完成操作后的返回路径
7. 数据是否保存、退出后是否恢复

默认要求：

- 先出交互图，再定实现
- 交互图确认后，作为后续开发基线
- 后续实现必须尽量与交互图保持一致
- 若实现时需偏离交互图，必须明确说明偏离点

## 7. 文档同步规则

项目推进过程中，凡形成以下任一结果时，必须同步更新对应项目文档：

- 新规则确认
- 新设计方案确认
- 模块阶段完成
- 版本交付完成
- 已确认方案变更
- 运行入口基线变更
- 当前在线壳切换

默认要求：

1. 代码更新后，相关文档也要同步更新
2. 文档内容必须与当前实现状态一致
3. 若实现与原方案有偏差，必须在文档中明确记录
4. 项目不能只改代码不留痕
5. 每个阶段结束后，要能通过文档快速恢复上下文

### 7.1 什么时候必须更新文档
出现下面这些情况时，默认更新：

- 某个需求被正式确认之后
- 一个模块完成一个阶段之后
- 新增或修改项目规则之后
- 已确认方案发生变更之后
- 一个版本交付之后
- 运行入口发生切换之后
- 发现“修改入口”和“运行入口”不一致之后

### 7.2 不需要每次都更新的情况
下面这些可以累计到阶段更新时一起写：

- 纯样式微调
- 文案小修改
- 不影响规则的小 bug 修复
- 不改变交互逻辑的小兼容性修复

原则：

**不是每改一个字都写文档，而是每形成阶段性结论，就更新文档。**

### 7.3 建议长期维护的项目文档
建议至少维护以下 4 类：

1. 项目总协作规范
2. 已确认设计方案汇总
3. 版本变更记录
4. 最新项目状态摘要

## 8. 默认优先级

### 第一优先级
把系统核心链路做完整、做稳定：

- 做题
- 错题沉淀
- 分类管理
- 复盘记录
- 数据保存恢复
- 页面交互顺手

### 第二优先级
把高频使用模块做好：

- 错题系统
- 知识库
- 画布
- 学习记录
- 统计分析

### 第三优先级
自动化能力：

- 自动记录学习过程
- 自动整理错因
- 自动生成每日小结
- 自动推送学习建议

### 第四优先级
AI增强能力

AI在系统骨架稳定后接入。当前阶段如非必要，不要让 AI 能力干扰主系统搭建。

## 9. 已确认基线提醒

### 9.1 画布功能 V1 基线
- 在题目卡片右上角增加“画布”按钮
- 与“展开”处于同一区域
- 点击后直接在题目内容区域上覆盖透明画布层
- 采用原地覆盖，不跳转页面
- 按钮与题目区域底色、风格保持统一
- 第一版工具仅保留：撤销 / 清空 / 退出
- 每道题的画布数据自动保存
- 再次打开同一道题时自动恢复之前内容

### 9.2 当前推进基线
- 当前阶段优先完成并优化系统本身
- 围绕项目时，优先给出完整方案、明确规则/边界/验收、按模块推进、每步自测并交付可用结果
- 涉及页面交互、布局、入口位置、弹层/覆盖/跳转关系等设计时，默认同步给出交互图或结构示意图
- 当前在线页面改造默认优先命中 `v51_frontend`

## 10. 默认禁止事项

- 不要只做局部，不交代整体位置
- 不要脱离原页面乱改布局
- 不要没说明边界就一次性乱扩展
- 不要只讲概念，不给可用结果
- 不要不自测就交付
- 不要忽略已确认方案
- 不要为了技术炫耀牺牲实用性
- 不要为了系统本身偏离提分目标
- 不要代码改了却不更新相关文档
- 不要交互变了却没有交互图或结构说明
- 不要在未确认运行入口前直接改前端页面
- 不要把“源码已改”误当成“当前页面已生效”


## 12. 新增经验：首版闭环与体验收敛要分阶段

以后涉及高频使用功能，默认区分两步：

1. 首版先打通行为闭环
2. 闭环稳定后再做体验收敛

对应到画布这次新增结论：
- 首版优先保证能画、能存、能恢复、不拖慢页面
- 第二步再收敛入口位置、按钮数量、颜色数量和保存方式
- 高频动作留在主工具条，低频动作移入二级菜单
- 用户做题时优先减少选择成本
