# v3.1 Integration Plan

## Goal

Absorb the downloaded v3.1 code into the current project without destabilizing the existing:

1. authenticated workbench flow
2. per-user full-backup storage
3. knowledge-tree-centered UI
4. local-first editing model

## Stability Rules

1. Keep the current full-backup flow working at every step.
2. Merge low-risk improvements before schema-changing features.
3. Treat image storage split and incremental sync as separate rollout phases.
4. Do not mix one-time data migrations with unrelated UI or AI changes.
5. Every absorbed change must have an explicit verification gate.

## Phase Order

### Phase 1: Safe absorption

Scope:

- tighten CORS from env instead of `*`
- remove debug-only user listing endpoint
- unify AI calling so DeepSeek can be used with MiniMax fallback
- improve analyze-entry prompt constraints
- make local error storage failures visible
- prevent cloud save from hanging forever
- expand frontend search coverage
- add mastery-level UI state
- flush pending cloud save on tab close / hide

Verification gate:

1. login and logout still work
2. cloud load and cloud save still work
3. analyze-entry still returns valid JSON
4. editing and deleting errors still work
5. page reload preserves local data

### Phase 2: Data model extension

Scope:

- add `user_images` table
- add image upload / read / unref APIs
- add orphan image cleanup script
- move toward file-backed image storage

Verification gate:

1. new image uploads deduplicate correctly
2. existing backups without split images still load
3. delete flow does not remove shared images early

### Phase 3: Incremental sync

Scope:

- add `operations` table
- add sync pull / push APIs
- add frontend op recording and replay
- add `updatedAt`-based merge semantics
- add integer-id to UUID migration

Verification gate:

1. old single-device flow still works
2. two-device edit conflict behavior is deterministic
3. migration runs once and does not corrupt revealed state
4. fallback to full cloud load remains available

## Deferred Risk Notes

These items stay deferred until Phase 1 is stable:

- one-time ID migration
- live incremental merge between devices
- image storage split away from browser backup payload
- background cleanup jobs

## Current Implementation Decision

The planned v3.1 mainline has now been absorbed into the codebase:

1. Phase 1 stability changes are implemented
2. image storage APIs and cleanup tooling are implemented
3. incremental sync APIs, UUID migration, and frontend op replay are implemented

## Residual Caveats

These are no longer blockers for v3.1 completion, but they remain important operating notes:

1. fine-grained incremental merge currently focuses on error records
2. note bodies / knowledge content still rely primarily on the full-backup path
3. full backup is intentionally kept as the rollback and restore safety net
4. public tunnel availability still depends on valid Cloudflare setup or a quick-tunnel fallback
