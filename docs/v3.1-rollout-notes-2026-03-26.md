# v3.1 Rollout Notes

## What We Standardized

1. Keep full-backup restore available even after adding incremental sync.
2. Prefer backend capability rollout before binding every new feature into the HTML UI.
3. For legacy `xingce_v3.html`, prefer CSS-led layout refresh over risky DOM surgery when encoding is brittle.
4. Public access should use a named Cloudflare tunnel and fixed domain, not a quick-tunnel URL.

## Layout Decisions

1. The related-errors right rail was removed from the main layout to keep the workspace centered.
2. The page now follows a simpler left-navigation + center-workspace model.
3. Content width is intentionally capped so long notes and grouped errors stay readable after removing the right rail.
4. The login page no longer exposes the current origin or tunnel URL.
5. Later layout polish should prefer CSS-led refinement before risky HTML surgery inside the legacy file.

## Deployment Decisions

1. `docker-compose.yml` now runs `cloudflared tunnel run --token ...` for stable named-tunnel mode.
2. `erroranaly.qzz.io` is bound to the named tunnel instead of rotating `trycloudflare.com` links.
3. `scripts/bind-cloudflare-domain.ps1` is the repeatable repair/bootstrap path for tunnel + DNS.
4. OCR now runs as a sidecar Docker service in the main stack instead of staying Tesseract-only forever.
5. the `umi-ocr` container must run in headless mode; GUI mode fails inside Docker without `DISPLAY`.

## API Rollout Decisions

The roadmap items that were still missing from code were first landed as backend capabilities:

1. daily practice queue
2. practice log persistence
3. evaluate-answer
4. generate-question
5. diagnose
6. chat
7. module summary for Claude handoff
8. distill / synthesize / discover-patterns / suggest-restructure
9. knowledge search

This keeps the implementation honest: the features now exist as callable capabilities even before each one gets polished UI affordances.

## OCR Rollout Decisions

1. keep `/api/ai/ocr-image` as the only frontend-facing OCR API
2. add env-driven backend routing inside `app/main.py`
3. prefer `OCR_BACKEND=umi` for the current Docker runtime
4. keep `OCR_TESSERACT_FALLBACK=true` for degraded-but-usable recovery
5. verify OCR through the real app runtime instead of treating the sidecar alone as success

## Verification We Actually Needed

This phase confirmed that the reliable acceptance sequence is:

1. local backend syntax check
2. local smoke check
3. Docker rebuild
4. sidecar OCR health check on `:1224`
5. local page smoke
6. public page smoke on `https://erroranaly.qzz.io`
7. public browser-context OCR upload returning `engine = "umi-ocr"`

## Lessons From Failed Assumptions

1. "Docker image exists" is not the same as "service is integrated into the app path"
2. "homepage returns 200" is not the same as "public OCR flow works"
3. direct public script requests can hit edge restrictions even when the browser flow is healthy
4. legacy duplicate function definitions in `app/main.py` can hide runtime behavior and must be checked before another OCR change

## Remaining Follow-Up

1. Connect the new practice and AI endpoints to first-class UI entry points.
2. Continue cleaning duplicated legacy CSS in `xingce_v3.html`.
3. Consider splitting the HTML file into modules once the UI shape stabilizes.
4. Build the Shenlun module on top of imported in-repo source materials instead of reading from external repo paths at runtime.
5. continue reducing legacy inline-script risk in `xingce_v3.html` and `app/main.py` where historical duplication still exists.
